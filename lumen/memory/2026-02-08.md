### Competitive Analysis: SQ vs LanceDB (10:20 CST)
Will asked: "How is SQ cleaner, faster, and smarter than vector search?" after seeing LanceDB pricing.

Created `/home/wbic16/.openclaw/workspace/exo-plan/competitive/SQ-vs-LanceDB.md` (9.3 KB):

**Key Findings:**
1. **Cleaner:** SQ = no embeddings, human-readable coordinates. LanceDB = 1536D float vectors, opaque.
2. **Faster:** SQ = O(1) coordinate lookup. LanceDB = O(log n) ANN search. (Caveat: SQ only fast for *exact lookups*, not semantic search.)
3. **Smarter:** SQ = structure as first-class (11D hierarchy). LanceDB = flat vector space.
4. **Pricing:** LanceDB = "usage-based" (hidden pricing, call sales). SQ = $50/mo transparent.

**Positioning:** LanceDB = semantic search engine. SQ = coordinate-based knowledge substrate. Different use cases. We compete on *transparency + data ownership*, not vector search performance.

**Questions for Will:**

**Update (10:25 CST):** Will added critical context — holographic storage (multiple coordinates per document) + multi-encoder API modes. Updated analysis:
